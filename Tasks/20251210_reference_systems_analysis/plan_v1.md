# План глубокого анализа референсных LMS-платформ

**Дата создания:** 2025-12-10
**Цель:** Детальный анализ 8 ведущих LMS-платформ для извлечения лучших практик проектирования B3 Learning Portal

---

## 1. Цель задачи

Провести детальный функциональный и UX/UI анализ каждой из 8 референсных образовательных платформ для формирования базы знаний проектной команды B3 LMS. Результат должен позволить:

- **Понять структуру данных** каждой платформы (основные сущности, связи)
- **Извлечь UX-паттерны** для адаптации в B3-специфичный контекст
- **Выявить ключевые бизнес-процессы** (BPMN-кандидаты для B3)
- **Сформировать список "что взять" и "что НЕ взять"** для каждой платформы
- **Создать сводку best practices** для быстрого доступа

---

## 2. Список платформ для анализа

Из документа `references.md` выделены следующие 8 платформ:

1. **Canvas LMS** (Instructure) - академическая, известна чистым UI
2. **Coursera** - глобальный MOOC, эталон прогресс-трекинга
3. **SAP Litmos** - корпоративная, простота + мощь
4. **Docebo LMS** - AI-powered, корпоративная с высоким UX
5. **D2L Brightspace** - академическая + корпоративная, обновленный UI
6. **Absorb LMS** - consumer-grade UX для бизнеса
7. **Moodle / Totara** - open-source, широчайшее распространение, обновленный UX в v4
8. **Udemy** - маркетплейс, эталон простоты входа и покупки курсов

## 3. Структура анализа каждой платформы

Для каждой платформы создается отдельный файл `NN_platform_name.md` со следующей структурой:

### 3.1. Краткий обзор (Overview)
- Тип системы (академическая/корпоративная/маркетплейс)
- Целевая аудитория
- Ключевые отличия от конкурентов
- Почему именно эта платформа в референсах

### 3.2. Структура системы (System Structure)
- **Основные роли:** студент, преподаватель, администратор, (другие)
- **Модель данных (сущности):**
  - Курс / Программа / Специализация
  - Задания / Тесты / Оценки
  - Пользователи / Группы / Когорты
  - Сертификаты / Достижения
  - Коммуникации (форумы, сообщения)
- **ER-диаграмма (текстовое описание):** связи между сущностями

### 3.3. UX/UI особенности (UX/UI Features)
- **Онбординг:** как новый пользователь попадает в систему
- **Навигация:** главное меню, дашборд, структура курса
- **Прогресс-трекинг:** как студент видит свой прогресс
- **Визуальный дизайн:** цветовая схема, типографика, UI-компоненты
- **Мобильная адаптация:** отзывчивость, нативные приложения
- **Уникальные UX-фишки:** что выделяет платформу

### 3.4. Ключевые процессы (Key Processes)
Описание основных сценариев использования в формате BPMN-like:

- **Регистрация и запись на курс**
- **Прохождение курса (student journey):**
  - Доступ к материалам
  - Выполнение заданий
  - Сдача тестов
  - Получение обратной связи
- **Взаимодействие с преподавателем:**
  - Вопросы / форумы
  - Проверка заданий
  - Выставление оценок
- **Завершение курса и сертификация:**
  - Условия получения сертификата
  - Формат сертификата
  - Место хранения
- **Административные процессы:**
  - Создание курса
  - Управление пользователями
  - Отчетность

### 3.5. Что можно взять для B3 LMS (Adopt for B3)
Конкретные элементы, которые:
- **Реализуемы средствами B3** (Data Model Constructor, BPMN Designer, Dashboard Builder)
- **Решают задачи учебного портала B3**
- **Улучшают UX по сравнению с типичными корпоративными системами**

Примеры:
- Паттерн "продолжить обучение" (кнопка на дашборде)
- Прогресс-бары на карточках курсов
- Структура сущностей "Шаблон курса → Экземпляр курса"
- BPMN-процесс автоматической выдачи сертификата

### 3.6. Что НЕ подходит (Don't Adopt)
Элементы, которые:
- **Требуют кастомной разработки вне B3**
- **Избыточны для корпоративного контекста B3**
- **Конфликтуют с ограничениями платформы B3**

Примеры:
- AI-генерация контента (требует внешних API)
- Peer-review механизмы (сложная логика, не первая очередь)
- Маркетплейс с онлайн-оплатой (не в scope v1.0)

### 3.7. Ссылки (References)
- Официальный сайт
- Документация (user guide, admin guide)
- API docs (если релевантно)
- Видеообзоры / демо
- Статьи / кейсы клиентов

## 5. Критерии качества анализа (Quality Criteria)

Каждый анализ платформы считается завершенным, если:

### 5.1. Полнота (Completeness)
- [ ] Все 7 секций заполнены (обзор, структура, UX, процессы, adopt, don't adopt, ссылки)
- [ ] Модель данных описана минимум 5 ключевых сущностей с указанием связей
- [ ] Описано минимум 3 ключевых процесса (регистрация, прохождение курса, сертификация)
- [ ] Список "что взять" содержит минимум 5 конкретных элементов
- [ ] Список "что НЕ брать" содержит минимум 3 обоснованных пункта

### 5.2. Релевантность (Relevance)
- [ ] Все выделенные элементы соотнесены с возможностями B3 Platform
- [ ] Приведены примеры реализации через B3 инструменты (Data Model, BPMN, Dashboard)
- [ ] Учтены ограничения B3 (нативные компоненты vs кастомный код)
- [ ] Есть ссылки на `gpt_design.md` при совпадении подходов

### 5.3. Практичность (Actionability)
- [ ] Описания UX содержат конкретные скриншоты или четкие текстовые описания экранов
- [ ] Процессы описаны в терминах "актор → действие → результат"
- [ ] Для каждого "adopt" элемента указан приоритет (must-have / nice-to-have)
- [ ] Для каждого "don't adopt" указана причина (техническая/бизнес/приоритет)

### 5.4. Критичность (Critical Thinking)
- [ ] Анализ содержит критику / ограничения платформы (не только хвалебный обзор)
- [ ] Указаны edge cases, которые платформа решает/не решает
- [ ] Есть сравнение с другими платформами (где уместно)
- [ ] Выделены "спорные" решения, требующие обсуждения с командой

### 5.5. Документированность (Documentation)
- [ ] Все ссылки рабочие и ведут на релевантные страницы
- [ ] Для каждой функции указан источник (документация / видео / собственный опыт)
- [ ] Скриншоты (если есть) сопровождены описанием
- [ ] Терминология согласована с `gpt_design.md` (русские названия сущностей)

## 6. Итоговый артефакт: `00_best_practices_summary.md`

После завершения всех 8 анализов создается сводный документ, содержащий:

### 6.1. Структура сводки

1. **Executive Summary**
   - Топ-10 UX-паттернов, которые ОБЯЗАТЕЛЬНО внедрить в B3 LMS
   - Топ-5 ошибок, которых избегать

2. **Сравнительная таблица платформ**
   - Матрица: платформа × ключевые функции (онбординг, прогресс-трекинг, сертификация, форумы, мобильность)
   - Оценка каждой функции (отлично / хорошо / средне / слабо / нет)

3. **Модель данных: универсальные паттерны**
   - Какие сущности есть во ВСЕХ платформах (курс, пользователь, задание, оценка)
   - Какие связи повторяются (Enrollment, Assignment Instance, Certificate)
   - Рекомендации для Data Model Constructor B3

4. **UX-паттерны: каталог решений**
   - **Дашборд студента:** 8 вариантов, лучший синтез
   - **Карточка курса:** обязательные элементы (название, прогресс, кнопка "продолжить", дедлайны)
   - **Страница курса:** структура модулей, навигация, боковое меню vs вкладки
   - **Прохождение задания:** от клика до получения оценки
   - **Сертификат:** формат, место выдачи, дизайн

5. **Процессы: BPMN-шаблоны для B3**
   - Процесс "Регистрация на курс" (3 варианта: свободная, по заявке, автоматическая)
   - Процесс "Жизненный цикл задания" (submit → review → grade → notify)
   - Процесс "Выдача сертификата" (условия → генерация PDF → отправка → хранение)
   - Процесс "Напоминание о дедлайнах" (автоматические уведомления)

6. **Технологический стек (референс)**
   - Какие технологии используют платформы (для понимания трендов)
   - Что из этого доступно в B3 out-of-the-box
   - Что потребует интеграций (например, видеоплееры, вебинары)

7. **Roadmap приоритизации**
   - MVP (v1.0): что внедрять в первую очередь
   - v2.0: расширенные функции
   - v3.0+: nice-to-have фичи

8. **Риски и подводные камни**
   - Что работает в SaaS-платформах, но сложно на B3
   - Где UX-ожидания пользователей могут не совпасть с возможностями B3
   - Как минимизировать разрыв

## 10. Формат файлов анализа

Каждый файл `NN_platform_name.md` должен:
- Быть на **русском языке**
- Использовать **Markdown** с заголовками H2/H3
- Включать **таблицы** для сравнений (роли, сущности)
- Содержать **списки** для процессов и рекомендаций
- **Ссылки** на источники в секции References

Пример начала файла:
```markdown
# Анализ: Canvas LMS

**Дата анализа:** 2025-12-10
**Аналитик:** [Имя агента/исполнителя]
**Источник:** https://www.instructure.com/canvas

---

## 1. Краткий обзор
Canvas LMS — облачная система управления обучением от компании Instructure...

[далее по структуре из раздела 3]
```
