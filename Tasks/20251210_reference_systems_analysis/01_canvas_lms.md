## 1. Краткий обзор

Canvas LMS — облачная система управления обучением от компании Instructure, широко применяемая в образовательных учреждениях и корпоративном секторе по всему миру. Canvas выделяется среди конкурентов своим **чистым, интуитивно понятным интерфейсом**, который облегчает навигацию как для студентов, так и для преподавателей.

### Тип системы
Академическая LMS с сильными корпоративными возможностями (Canvas for Business).

### Целевая аудитория
- Университеты и колледжи (основной сегмент)
- Школы K-12
- Корпоративное обучение
- Государственные учреждения

### Ключевые отличия от конкурентов

1. **UX/UI как главное конкурентное преимущество** — студенты особенно ценят мобильный доступ, понятное отслеживание прогресса и интуитивный интерфейс. Опросы показывают, что Canvas считается одним из самых дружелюбных LMS в отрасли.

2. **Облачная архитектура** — полностью SaaS-решение, не требует локальной установки, автоматические обновления, масштабируемость.

3. **Богатая экосистема интеграций** — встроенная поддержка LTI (Learning Tools Interoperability), интеграции с Zoom, Google Drive, Microsoft Teams и сотнями других сервисов.

4. **Открытый REST API** — все данные и функции доступны через документированный API, что позволяет строить кастомные интеграции и автоматизацию.

5. **Мобильные приложения высокого качества** — нативные iOS/Android приложения с рейтингом 4.7/4.6 звезд и более 10 млн загрузок.

### Почему Canvas в референсах для B3 LMS

Canvas выбран как эталонный пример того, как **академическая система может быть одновременно мощной и простой в использовании**. Для российских корпоративных и государственных проектов Canvas интересен тем, что:

- Его подход к дизайну масштабируем и понятен широкой аудитории (от школьников до топ-менеджеров)
- Успешно применяется в больших организациях (университетские сети с десятками тысяч студентов)
- Демонстрирует, как можно обеспечить богатую функциональность без ущерба для простоты интерфейса
- Показывает best practices для студентского дашборда, навигации по курсу, системы оценивания

## 2. Структура системы

### 2.1. Основные роли

Canvas поддерживает следующие роли с четким разделением прав доступа:

| Роль | Английское название | Описание |
|------|---------------------|----------|
| **Студент** | Student | Просмотр материалов курса, выполнение заданий, сдача тестов, участие в дискуссиях, просмотр оценок |
| **Преподаватель** | Teacher / Instructor | Создание и редактирование курсов, загрузка контента, создание заданий/тестов, проверка работ через SpeedGrader, выставление оценок, коммуникация со студентами |
| **Ассистент преподавателя** | TA (Teaching Assistant) | Помощь в проверке заданий, модерация дискуссий, ограниченные права на редактирование курса |
| **Дизайнер курса** | Designer | Создание и настройка структуры курса, загрузка контента, настройка модулей (без доступа к оценкам студентов) |
| **Наблюдатель** | Observer | Просмотр активности конкретного студента (обычно для родителей), доступ только на чтение |
| **Администратор** | Admin | Управление учетными записями, настройка системы, управление курсами на институциональном уровне, аналитика |

### 2.2. Модель данных (сущности)

Canvas построен на реляционной модели данных с четкой иерархией сущностей:

#### Основные сущности

**1. Account (Учетная запись / Организация)**
- Корневая сущность для институции
- Определяет права доступа, настройки, брендинг
- Может иметь подаккаунты (sub-accounts) для департаментов

**2. User (Пользователь)**
- Уникальный ID пользователя в системе
- Профиль: имя, email, аватар, часовой пояс
- Может иметь SIS ID (Student Information System ID) для интеграции с внешними системами
- Связан с множественными Enrollments

**3. Course (Курс)**
- Атрибуты: название, код курса, описание, силлабус
- Временные рамки: `created_at`, `start_at`, `end_at`
- Статус: опубликован / не опубликован / завершен / удален
- `default_view`: какой вид курса показывается по умолчанию (modules, assignments, syllabus и т.д.)
- `total_students`: количество записанных студентов
- Поддержка `locale` для многоязычности
- Связь с Enrollment Term (учебный семестр/период)

**4. Enrollment (Запись на курс)**
- Связь многие-ко-многим между User и Course
- Атрибуты: `role_id` (тип записи), `enrollment_state` (active, invited, completed, inactive)
- Временные метки: `created_at`, `updated_at`, `start_at`, `end_at` (ISO8601 формат)
- Типы: StudentEnrollment, TeacherEnrollment, TaEnrollment, DesignerEnrollment, ObserverEnrollment
- Для StudentEnrollment хранятся данные грейдинга: `current_grade`, `final_grade`, `current_score`, `final_score`

**5. Module (Модуль курса)**
- Организационная единица внутри курса
- Содержит упорядоченный список Module Items
- Поддержка prerequisites (пререквизитов) — модуль можно открыть только после выполнения предыдущего
- Отслеживание completion (завершения)

**6. Assignment (Задание)**
- Ключевая сущность для оценочных активностей
- Типы submission (сдачи): file upload (загрузка файла), text entry (текстовый ввод), URL, media, external tool
- Атрибуты: название, описание, дедлайн (`due_at`), максимальный балл (`points_possible`)
- Связь с Rubric (рубрикой оценивания) — опционально
- Поддержка дифференцированных дедлайнов для разных студентов/групп
- `grading_period_data`: связь с периодами оценивания

**7. Submission (Сдача задания)**
- Связь между Assignment и Student
- `workflow_state`: статус (submitted, graded, pending_review, unsubmitted)
- `submitted_at`: timestamp сдачи
- `grade`: оценка (строка — может быть баллы, процент, буквенная оценка)
- `score`: числовое значение балла
- Вложенные файлы / текст / URL в зависимости от типа
- История попыток (attempts) — Canvas хранит все версии сдач

**8. Discussion (Дискуссия / Форум)**
- Асинхронное общение внутри курса
- Типы: announcement (объявление, только для чтения), threaded discussion (древовидная дискуссия), graded discussion (оцениваемая дискуссия)
- Поддержка вложений, rich-text форматирования

**9. Quiz (Тест / Викторина)**
- Различные типы вопросов: multiple choice, true/false, fill-in-the-blank, essay, matching, numerical и др.
- Автоматическая проверка для объективных вопросов
- Настройки: ограничение по времени, количество попыток, показ правильных ответов
- Интеграция с Gradebook

**10. Gradebook (Журнал оценок)**
- Матрица: студенты × задания
- Автоматический расчет итоговых оценок на основе весов категорий
- Поддержка различных схем оценивания (points, percentage, letter grade, GPA scale)
- Экспорт в CSV для внешней обработки

**11. Page (Страница контента)**
- Статический контент курса (лекции, материалы)
- Rich-text редактор с поддержкой встроенного видео, изображений
- Версионирование — история изменений

**12. File (Файл)**
- Хранилище файлов курса / пользователя
- Организация в папки
- Права доступа: публичный / только для курса / скрытый
- Поддержка preview для распространенных форматов

**13. Group (Группа)**
- Организация студентов внутри курса для групповой работы
- Групповые задания (group assignments) — одна сдача от группы, оценка всем участникам
- Групповые дискуссии и файловое пространство

**14. Calendar Event (Событие календаря)**
- Интеграция с дедлайнами заданий
- Синхронизация с внешними календарями (Google Calendar, Outlook)
- Личные и курсовые события

**15. Notification (Уведомление)**
- Система оповещений о новых оценках, комментариях, объявлениях
- Настройка каналов доставки: email, SMS, push в мобильных приложениях
- Пользователь может настроить частоту и типы уведомлений

#### Интеграционные сущности

**16. SIS ID (Student Information System ID)**
- Canvas позволяет хранить уникальные идентификаторы из внешних систем (ERP, HR-системы)
- Используется для синхронизации данных о пользователях, курсах, записях через API
- Упрощает интеграцию с институциональными системами

**17. External Tool (LTI Integration)**
- Интеграция сторонних приложений через стандарт LTI (Learning Tools Interoperability)
- Примеры: видеоплатформы, системы прокторинга экзаменов, специализированные симуляторы
- Поддержка SSO (Single Sign-On) для бесшовного входа

### 2.3. ER-диаграмма (текстовое описание связей)

```
Account (1) ──< (M) Sub-Account
Account (1) ──< (M) Course
Account (1) ──< (M) User

User (M) ──< Enrollment >── (M) Course
   ↓ (role_id: StudentEnrollment, TeacherEnrollment, etc.)

Course (1) ──< (M) Module
Course (1) ──< (M) Assignment
Course (1) ──< (M) Discussion
Course (1) ──< (M) Quiz
Course (1) ──< (M) Page
Course (1) ──< (M) File
Course (1) ──< (M) Group

Module (1) ──< (M) ModuleItem (ссылка на Assignment / Page / Discussion / Quiz / File / ExternalTool)

Assignment (1) ──< (M) Submission
Submission (M) ──> (1) User (студент)
Submission (1) ──< (M) SubmissionComment (от преподавателя или студента)

Assignment (1) ──> (0..1) Rubric (критерии оценивания)

User (1) ──< (M) CalendarEvent
Course (1) ──< (M) CalendarEvent

User (1) ──< (M) Notification

Course (1) ──< (M) Enrollment ──> (1) EnrollmentTerm (семестр)
```

**Ключевые связи для B3 LMS:**

- **Шаблон vs Экземпляр:** Canvas не разделяет явно "шаблон курса" и "экземпляр курса" — вместо этого есть механизм **Course Copy** (копирование курса). Преподаватель создает master-курс, затем копирует его для нового семестра/потока. Для B3 это означает, что паттерн "Шаблон → Экземпляр" нужно реализовывать явно в модели данных.

- **Enrollment как центральная сущность:** связь User ↔ Course через Enrollment с атрибутом role обеспечивает гибкость — один пользователь может быть студентом одного курса и преподавателем другого.

- **Gradebook вычисляется динамически:** нет отдельной таблицы "итоговые оценки" — они рассчитываются на лету из Submissions + настроек весов категорий.

## 3. UX/UI особенности

### 3.1. Онбординг: как новый пользователь попадает в систему

**Для студентов:**

1. **Регистрация / приглашение**
   - Самостоятельная регистрация (если разрешена институцией) через форму
   - Приглашение по email от администратора / преподавателя
   - Интеграция с SSO (Single Sign-On) через SAML / OAuth — студент входит через корпоративный аккаунт

2. **Первый вход**
   - Заполнение профиля (фото, часовой пояс, notification preferences)
   - Опциональный tour по интерфейсу (Welcome to Canvas)
   - Автоматическая запись на курсы (если enrollment уже создан администратором)

3. **Dashboard как стартовая точка**
   - После входа студент сразу видит Dashboard с карточками курсов
   - Минимум кликов до начала обучения: Dashboard → Курс → Задание

**Для преподавателей:**

1. **Setup wizard при создании первого курса**
   - Выбор default view (модули / задания / страницы)
   - Импорт контента (из предыдущих курсов / внешних источников)
   - Настройка навигации и home page

2. **Шаблоны курсов (Course Templates)**
   - Институция может предоставить преподавателям готовые шаблоны с типовой структурой
   - Canvas Commons — публичная библиотека курсов, которые можно импортировать

### 3.2. Навигация: главное меню, дашборд, структура курса

**Global Navigation (Глобальная навигация — левая панель)**

Canvas использует постоянную вертикальную панель слева, доступную на всех страницах:

- **Account (иконка учреждения)** — настройки профиля, уведомлений
- **Dashboard** — главная страница пользователя
- **Courses** — список всех курсов (активных и завершенных)
- **Calendar** — агрегированный календарь всех курсов с дедлайнами
- **Inbox** — внутренняя система сообщений (как email внутри Canvas)
- **Commons** — библиотека общедоступных курсов (если включена)
- **Help** — ссылка на документацию, support

Эта панель **контекстно-независима** — она есть везде, что снижает когнитивную нагрузку (студент всегда знает, как вернуться на Dashboard или открыть Calendar).

**Student Dashboard (Дашборд студента)**

Три режима отображения (студент выбирает в настройках):

1. **Card View (Карточный вид)** — по умолчанию
   - Крупные карточки для каждого курса
   - На карточке: название курса, код, изображение (course image), nickname (если задан)
   - Кнопка "Star" для добавления в избранное (favorite courses)
   - Клик по карточке → переход на home page курса

2. **List View (Список)**
   - Компактный список курсов с To-Do items (ближайшими заданиями)
   - Agenda-формат: задания по датам

3. **Recent Activity View (Недавняя активность)**
   - Лента событий: новые оценки, комментарии, объявления

**Правая боковая панель Dashboard:**
- **To Do** — список ближайших заданий (across all courses)
- **Coming Up** — события календаря на ближайшую неделю
- **Recent Feedback** — последние комментарии от преподавателей

**UX-инсайт:** Canvas фокусируется на **контексте "что делать дальше"**, а не "где я нахожусь" — кнопка "продолжить" всегда ведет к следующему шагу.

**Course Navigation (Навигация внутри курса)**

После входа в курс студент видит:

- **Верхняя панель:** название курса, breadcrumbs (навигационные крошки)
- **Левое меню курса (Course Navigation Menu):**
  - Home (главная страница курса)
  - Announcements (объявления)
  - Assignments (список заданий)
  - Discussions (форумы)
  - Grades (оценки студента)
  - People (участники курса)
  - Pages (страницы контента)
  - Files (файлы)
  - Syllabus (силлабус / программа курса)
  - Outcomes (образовательные результаты / компетенции)
  - Quizzes (тесты)
  - Modules (модули) — **самый популярный режим навигации**
  - Collaborations (совместная работа — Google Docs интеграция)
  - Settings (настройки курса — только для преподавателей)

Преподаватель может **кастомизировать** это меню: скрыть ненужные пункты, изменить порядок, переименовать (например, "Modules" → "Недели курса").

**Modules — ключевой паттерн структурирования курса:**

- Модуль = логический блок курса (неделя, тема, раздел)
- Внутри модуля — упорядоченный список элементов: Pages, Assignments, Quizzes, Discussions, External Tools, Files
- Поддержка **prerequisites** — модуль 2 откроется только после завершения модуля 1
- Отображение completion status: галочки рядом с каждым элементом (done / not done)
- Кнопка **"Next"** в конце каждого элемента модуля → автоматический переход к следующему элементу
- Студент видит прогресс по модулю: "3 of 8 completed"

### 3.3. Прогресс-трекинг: как студент видит свой прогресс

Canvas реализует **многоуровневую систему прогресса:**

**1. Уровень дашборда:**
- На карточках курсов нет встроенного progress bar (это критика со стороны пользователей — см. раздел 6)
- Студент видит количество непрочитанных уведомлений / невыполненных заданий через badges (красные кружочки с цифрами)

**2. Уровень курса — Grades (оценки):**
- Вкладка **Grades** в курсе показывает таблицу: задание | статус | балл | feedback
- Цветовая индикация:
  - **Серый** — задание не сдано (missing)
  - **Оранжевый** — задание сдано, ожидает проверки
  - **Зеленый** — задание проверено, оценка выставлена
- Итоговая оценка курса (вверху страницы): current score / final score
- График изменения оценки во времени (grade distribution chart)

**3. Уровень модулей:**
- Рядом с каждым элементом модуля — иконка completion:
  - Пустой кружок — не начато
  - Галочка в кружке — завершено
  - Для некоторых элементов — автоматическое отслеживание (просмотрел страницу → автоматически "done")
- Прогресс-бар модуля: "5 of 12 items completed"

**4. Календарь и To-Do:**
- **Calendar** показывает все дедлайны на временной шкале (месяц / неделя / день)
- **To-Do list** на Dashboard — динамический список: задания сортируются по дедлайну, выполненные исчезают
- Интеграция с внешними календарями (Google, Outlook) через iCal feed

**5. Progress Reports (отчеты о прогрессе):**
- Преподаватель может включить модуль **Student Progress** — студент видит детализацию:
  - Время, проведенное в курсе
  - Количество просмотренных страниц
  - Participation score (активность в дискуссиях)

**Что отсутствует (и важно для B3 LMS):**
- Нет визуального прогресс-бара на карточке курса на Dashboard (только через кастомные расширения)
- Нет геймификации out-of-the-box (нет встроенных badges / achievements) — только через сторонние интеграции (Canvas Credentials, Badgr)

### 3.4. Визуальный дизайн: цветовая схема, типографика, UI-компоненты

**Цветовая схема:**
- **Primary color:** синий (#008EE2 — Instructure blue)
- **Accent colors:** настраиваются институцией (branding)
- **Semantic colors:**
  - Зеленый — success (задание проверено, оценка положительная)
  - Оранжевый — warning (дедлайн скоро, задание на проверке)
  - Красный — error (задание просрочено, низкая оценка)
  - Серый — neutral (задание не начато)

**Типографика:**
- Шрифт: **Lato** (sans-serif) — чистый, читаемый
- Иерархия: четкое разделение заголовков (H1-H6), body text, captions

**UI-компоненты:**

## 4. Ключевые процессы

### 4.1. Регистрация и запись на курс

**Процесс 1: Самостоятельная регистрация (self-enrollment)**

```
[Студент] → Открывает публичную ссылку на курс → [Canvas: страница курса]
    ↓
Canvas проверяет: self-enrollment разрешен?
    ↓ (да)
[Студент] → Кнопка "Enroll" → Заполнение базовой формы (если требуется)
    ↓
Canvas создает: User (если новый) + Enrollment (role=StudentEnrollment, state=active)
    ↓
[Студент] → Автоматический редирект на Dashboard → Курс появляется в списке
    ↓
Canvas отправляет: Welcome email + добавляет в Calendar дедлайны курса
```

**Процесс 2: Запись по приглашению администратора**

```
[Администратор] → Canvas Admin Panel → People → Add Users
    ↓
Вводит список email / загружает CSV (bulk enrollment)
    ↓
Canvas создает: User (если новый) + Enrollment (state=invited)
    ↓
Canvas отправляет: Invitation email со ссылкой активации
    ↓
[Студент] → Кликает ссылку → Устанавливает пароль (если новый пользователь)
    ↓
Enrollment.state = active
    ↓
[Студент] → Видит курс на Dashboard
```

**Процесс 3: Интеграция с SIS (Student Information System)**

```
[Внешняя система (SIS/HR)] → Canvas SIS Import API
    ↓
POST /api/v1/accounts/:account_id/sis_imports (CSV file)
    ↓
Canvas парсит CSV со столбцами: user_id, course_id, role, status
    ↓
Canvas создает/обновляет: Users, Courses, Enrollments
    ↓
Canvas возвращает: Import report (success/failed records)
    ↓
[Администратор] → Проверяет лог импорта, исправляет ошибки при необходимости
```

**Бизнес-правила:**
- Enrollment может иметь `start_at` и `end_at` — студент видит курс только в этот период
- Enrollment можно деактивировать (state=inactive) — студент теряет доступ, но данные сохраняются
- Enrollment можно удалить (state=deleted) — студент исчезает из списка курса, но submissions сохраняются для истории

### 4.2. Прохождение курса (student journey)

#### 4.2.1. Доступ к материалам

```
[Студент] → Dashboard → Клик на карточку курса → [Canvas: Course Home Page]
    ↓
Canvas отображает: Home page (настройка преподавателя: Modules / Assignments / Syllabus / Pages)
    ↓ (типичный случай: Modules)
[Студент] → Видит список модулей с элементами
    ↓
Canvas показывает: статус каждого элемента (locked / available / completed)
    ↓
[Студент] → Кликает на элемент (Page / Assignment / Quiz / Discussion)
    ↓
Canvas отображает: контент элемента + кнопка "Mark as done" (если настроено manual completion)
    ↓
[Студент] → Читает/просматривает материал → Кликает "Mark as done"
    ↓
Canvas обновляет: ModuleItem.completion_status = completed
    ↓
Canvas проверяет: все элементы модуля завершены?
    ↓ (да)
Canvas разблокирует: следующий модуль (если установлен prerequisite)
```

**Типы материалов:**
- **Pages:** статический контент (лекции, инструкции) — открываются в том же окне
- **Files:** PDF, презентации — открываются в Canvas DocViewer или скачиваются
- **External URLs:** ссылки на внешние ресурсы (документация, видео на YouTube)
- **External Tools (LTI):** встроенные фреймы с интерактивным контентом (симуляторы, лабораторные работы)

#### 4.2.2. Выполнение заданий

```
[Студент] → Открывает Assignment из Modules или из списка Assignments
    ↓
Canvas отображает Assignment page:
   - Название, описание, дедлайн (due date)
   - Максимальный балл (points possible)
   - Submission type (file upload / text entry / URL / external tool)
   - Прикрепленные файлы от преподавателя (instructions, templates)
   - Рубрика (если есть) — показывает критерии оценивания
    ↓
[Студент] → Готовит решение (файл / текст / ссылка)
    ↓
[Студент] → Кликает "Submit Assignment" → Загружает файл или вводит текст в форму
    ↓
Canvas создает: Submission (workflow_state=submitted, submitted_at=timestamp)
    ↓
Canvas отправляет: уведомление преподавателю (если настроено)
    ↓
Canvas обновляет: Assignment page студента → показывает "Submitted" badge + timestamp
    ↓
[Студент] может: просмотреть свою сдачу, добавить комментарий, пересдать (если разрешены multiple attempts)
```

**Типы submission:**
- **Online upload:** студент загружает файл (PDF, Word, изображение, архив)
- **Text entry:** студент вводит текст в rich-text редактор прямо в Canvas
- **Website URL:** студент дает ссылку (например, на GitHub repo, Google Doc)
- **Media recording:** студент записывает видео/аудио через веб-камеру прямо в Canvas
- **External tool:** выполнение задания в интегрированном приложении (например, кодинг в CodeHS, дизайн в Figma)

**Особенности:**
- **Multiple attempts:** преподаватель может разрешить несколько попыток — Canvas сохраняет все версии, преподаватель видит историю
- **Group assignments:** одна сдача от группы, оценка проставляется всем участникам автоматически
- **Late submissions:** если студент сдал после дедлайн, Canvas помечает submission как "late" (красный badge), но принимает — преподаватель решает, штрафовать ли

#### 4.2.3. Сдача тестов

```
[Студент] → Открывает Quiz из модуля или списка Quizzes
    ↓
Canvas отображает Quiz instructions:
   - Количество вопросов, время (если ограничено)
   - Количество попыток
   - Политика показа правильных ответов (immediately / after due date / never)
    ↓
[Студент] → Кликает "Begin Quiz"
    ↓
Canvas запускает: таймер (если установлен time limit), блокирует другие вкладки (если настроен Lockdown Browser)
    ↓
Canvas отображает: вопросы (по одному или все сразу — настройка преподавателя)
    ↓
[Студент] → Отвечает на вопросы
    ↓
[Студент] → Кликает "Submit Quiz"
    ↓
Canvas сохраняет: QuizSubmission (timestamp, answers)
    ↓
Canvas автоматически проверяет: objective questions (multiple choice, true/false, matching, numerical)
    ↓
Canvas вычисляет: score (для автопроверяемых вопросов)
    ↓
Canvas создает: Submission в Gradebook (связь Quiz → Assignment → Submission)
    ↓
Canvas отображает: результаты студенту (если настройка разрешает)
   - Итоговый балл
   - Правильные/неправильные ответы (если показ разрешен)
   - Feedback по каждому вопросу (если настроен преподавателем)
```

**Типы вопросов:**
- **Автопроверяемые:** multiple choice, true/false, multiple answers, fill-in-the-blank, matching, numerical, formula (математические задачи)
- **Ручная проверка:** essay (эссе) — студент вводит текст, преподаватель проверяет вручную в SpeedGrader
- **File upload question:** студент загружает файл как ответ на вопрос теста

**Прокторинг:**
- Canvas интегрируется с системами прокторинга (Proctorio, Respondus Lockdown Browser, Honorlock)
- Можно включить: webcam monitoring, screen recording, browser lockdown (запрет на переключение вкладок)

#### 4.2.4. Получение обратной связи

```
[Преподаватель] → Открывает SpeedGrader для Assignment
    ↓
SpeedGrader отображает: submission студента (файл / текст / URL)
    ↓
[Преподаватель] → Оставляет комментарии (text / audio / video / annotations на PDF)
    ↓
[Преподаватель] → Выставляет оценку (grade) или отмечает критерии Rubric
    ↓
Canvas обновляет: Submission.grade, Submission.workflow_state = graded
    ↓
Canvas пересчитывает: Gradebook (итоговую оценку курса, если задание входит в общий балл)
    ↓
Canvas отправляет: уведомление студенту (email / push notification) "Your assignment has been graded"
    ↓
[Студент] → Получает уведомление → Открывает Assignment
    ↓
Canvas отображает:
   - Выставленную оценку (X out of Y points)
   - Комментарии преподавателя (текст / audio player / video player)
   - Аннотации на файле (если были)
   - Заполненную рубрику (с отмеченными критериями и комментариями)
    ↓
[Студент] → Может ответить на комментарий (двусторонняя коммуникация в thread)
    ↓
Canvas создает: SubmissionComment (author=student)
    ↓
Canvas уведомляет: преподавателя о новом комментарии
```

**Типы feedback:**
- **Текстовые комментарии:** общий комментарий к работе + inline-комментарии на конкретные строки кода/текста
- **Audio feedback:** преподаватель записывает голосовой комментарий (удобно для объяснения сложных моментов)
- **Video feedback:** преподаватель записывает видео (screen capture + камера) с разбором работы
- **Rubric:** структурированная обратная связь по критериям
- **Annotations:** пометки, подчеркивания, стрелки на PDF/документах студента

### 4.3. Взаимодействие с преподавателем

#### 4.3.1. Вопросы / форумы

**Сценарий 1: Дискуссионный форум курса**

```
[Преподаватель] → Создает Discussion topic в курсе
    ↓
Настраивает: тип (announcement / discussion / graded discussion), дедлайн (если graded)
    ↓
Canvas публикует: Discussion в курсе
    ↓
[Студент] → Видит Discussion в модуле или в списке Discussions
    ↓
[Студент] → Открывает → Читает вопрос/тему
    ↓
[Студент] → Пишет reply (ответ) — text + attachments + embedded media
    ↓
Canvas сохраняет: DiscussionEntry (автор, текст, timestamp)
    ↓
Canvas уведомляет: преподавателя + других участников (если подписаны на тему)
    ↓
[Преподаватель] → Отвечает на вопрос студента (reply to entry)
    ↓
Canvas создает: вложенный DiscussionEntry (threaded discussion — древовидная структура)
    ↓
[Другие студенты] → Могут ответить на пост студента (peer discussion)
    ↓
(если Discussion graded)
[Преподаватель] → Оценивает качество участия студента в дискуссии через SpeedGrader
```

**Типы дискуссий:**
- **Announcement (объявление):** односторонняя коммуникация (только преподаватель пишет, студенты читают)
- **Ungraded discussion:** свободное обсуждение, не влияет на оценку
- **Graded discussion:** участие оценивается (количество постов, качество, своевременность)

**Особенности:**
- Threaded discussions — вложенные ответы (как в Reddit)
- Поддержка упоминаний (@username) → уведомление пользователю
- Возможность like (лайкнуть) пост
- Поиск по дискуссиям

#### 4.3.2. Проверка заданий

(Описано в разделе 4.2.4 — процесс SpeedGrader)

**Дополнительные сценарии:**

**Массовая проверка тестов:**
```
[Преподаватель] → Открывает Quiz → Moderate Quiz
    ↓
Canvas отображает: список всех студентов + их попытки
    ↓
[Преподаватель] → Видит автоматически проверенные вопросы (уже оценены)
    ↓
[Преподаватель] → Кликает на студента с essay-вопросами → SpeedGrader
    ↓
Проверяет essay, выставляет баллы вручную
    ↓
Canvas обновляет: QuizSubmission.score = auto_score + manual_score
```

**Regrade (переоценка):**
```
[Преподаватель] → Обнаружил ошибку в вопросе теста (неправильный правильный ответ)
    ↓
[Преподаватель] → Редактирует вопрос → Canvas: "Regrade quiz?"
    ↓
[Преподаватель] → Выбирает стратегию:
   - Current and past attempts (пересчитать все сдачи)
   - Current attempts only (только новые)
    ↓
Canvas пересчитывает: scores всех студентов автоматически
    ↓
Canvas уведомляет: студентов об изменении оценок
```

#### 4.3.3. Выставление оценок

**Процесс: Gradebook (журнал оценок)**

```
[Преподаватель] → Grades → Gradebook (матрица студенты × задания)
    ↓
Canvas отображает:
   - Строки: студенты
   - Столбцы: assignments, quizzes, discussions (всё, что оценивается)
   - Ячейки: баллы или статусы (missing, late, graded)
    ↓
[Преподаватель] → Может вручную редактировать оценки (double-click на ячейку)
    ↓
Canvas автоматически вычисляет: итоговую оценку курса (Total column)
   - На основе весов категорий (Assignments 40%, Quizzes 30%, Participation 30%)
   - Политика dropped scores (например, "drop lowest 2 quiz scores")
    ↓
[Преподаватель] → Может настроить grading scheme:
   - Points (баллы)
   - Percentage (процент)
   - Letter grades (A, B, C, D, F) с пороговыми значениями
   - GPA scale (4.0, 3.7, 3.3, ...)
    ↓
Canvas отображает: итоговую оценку студентам в их Grades view
```

**Политики оценивания:**
- **Muted assignments:** оценки выставлены, но скрыты от студентов до момента unmute (публикации)
- **Dropped scores:** автоматическое исключение N худших оценок из категории (например, "drop 2 lowest quizzes")
- **Late policies:** автоматическое снижение балла за просроченные задания (например, -10% за каждый день просрочки, максимум 50%)
- **Missing submissions:** автоматическая 0 за несданные задания или опция "treat ungraded as 0"

**Экспорт и импорт:**
- Canvas позволяет экспортировать Gradebook в CSV
- Преподаватель может редактировать оценки в Excel, затем импортировать обратно

### 4.4. Завершение курса и сертификация

**Процесс: Завершение курса**

```
[Студент] → Выполняет последнее задание курса
    ↓
Canvas обновляет: последний ModuleItem.completion_status = completed
    ↓
Canvas проверяет: все обязательные элементы курса завершены?
    ↓ (да)
Canvas обновляет: Enrollment.workflow_state = completed
    ↓
Canvas вычисляет: final_grade студента (итоговая оценка с учетом всех политик)
    ↓
Canvas отображает: студенту в Grades view финальную оценку + статус "Completed"
```

**Важно:** Canvas из коробки НЕ выдает сертификаты автоматически. Это требует интеграции с внешними сервисами.

**Процесс: Выдача сертификата (через интеграцию с Canvas Credentials / Accredible / Badgr)**

```
[Преподаватель] → Настраивает интеграцию Canvas Credentials в курсе
    ↓
Создает: Badge template (шаблон бейджа/сертификата)
   - Дизайн (изображение, логотип)
   - Критерии выдачи (final_grade >= 70%, completion of all modules)
    ↓
Canvas Credentials автоматически отслеживает: прогресс студентов
    ↓
Когда студент выполняет критерии:
    ↓
Canvas Credentials генерирует: Digital Badge (открытый стандарт Open Badges)
    ↓
Canvas Credentials отправляет: уведомление студенту + email с ссылкой на бейдж
    ↓
[Студент] → Получает цифровой бейдж:
   - Верифицируемый (с метаданными: кто выдал, за что, когда)
   - Shareable (можно добавить в LinkedIn, резюме, портфолио)
   - Хранится в Canvas Credentials backpack (личная коллекция достижений)
    ↓
[Студент] → Может скачать: certificate PDF (если шаблон настроен)
```

**Альтернативные подходы к сертификации:**

1. **Ручная выдача:**
   - Преподаватель вручную генерирует PDF-сертификаты (вне Canvas)
   - Загружает их как Files в курс
   - Отправляет ссылку студентам через Announcements

2. **Интеграция с внешними системами:**
   - **Accredible:** автоматическая генерация сертификатов и бейджей на основе Canvas Gradebook
   - **Credly / Badgr:** управление цифровыми бейджами
   - **Parchment:** официальные академические транскрипты

3. **Кастомная автоматизация через Canvas API:**
   - Скрипт отслеживает Enrollment.workflow_state через API
   - При `completed` + `final_score >= threshold` → вызов внешнего API для генерации PDF-сертификата
   - Загрузка сертификата обратно в Canvas через Files API

**Что важно для B3 LMS:**
- Canvas Credentials — это **платный add-on** (не входит в базовую лицензию)
- Для корпоративного контекста в России критична автоматическая генерация PDF-сертификатов с QR-кодом для верификации
- B3 Platform уже имеет **конструктор печатных форм** — это преимущество перед Canvas, где сертификаты требуют интеграции

### 4.5. Административные процессы

#### 4.5.1. Создание курса

```
[Преподаватель/Designer] → Canvas → Courses → + Course
    ↓
Заполняет форму:
   - Course name, course code
   - Term (enrollment term — семестр)
   - Start/end dates (опционально)
    ↓
Canvas создает: Course (workflow_state=unpublished)
    ↓
[Преподаватель] → Настраивает Course Settings:
   - Default view (Modules / Assignments / Syllabus)
   - Navigation menu (какие пункты показывать)
   - Grading scheme
   - Course image (для карточки на Dashboard)
    ↓
[Преподаватель] → Создает контент:
   - Modules → добавляет Pages, Assignments, Quizzes, Discussions
   - Настраивает prerequisites, completion requirements
    ↓
[Преподаватель] → Публикует курс (кнопка "Publish")
    ↓
Canvas обновляет: Course.workflow_state = available
    ↓
Студенты, enrolled в курсе, видят его на Dashboard
```

**Ускорение создания курса:**
- **Course Copy:** преподаватель копирует прошлогодний курс, обновляет даты и материалы
- **Course Templates:** институция предоставляет готовые шаблоны (например, "Стандартный курс с 12 неделями")
- **Canvas Commons:** импорт публичных курсов из библиотеки

#### 4.5.2. Управление пользователями

**Массовое добавление пользователей (bulk enrollment):**

```
[Администратор] → Admin Panel → Account → Users → Add Users
    ↓
Выбирает метод:
   1. Manual entry (вручную ввести email/имена)
   2. CSV upload (загрузить CSV-файл)
   3. SIS import (синхронизация с внешней системой)
    ↓
(если CSV upload)
[Администратор] → Загружает CSV со столбцами: user_id, login_id, first_name, last_name, email, status
    ↓
Canvas парсит CSV → создает User records
    ↓
Canvas отправляет: invitation emails новым пользователям
    ↓
(если SIS import)
[Администратор] → Настраивает SIS integration:
   - API endpoint внешней системы
   - Mapping полей (external_id → Canvas SIS ID)
   - Расписание синхронизации (ежедневно / по событию)
    ↓
Canvas автоматически: создает/обновляет/деактивирует Users на основе данных SIS
```

**Управление ролями:**
- Администратор может создавать кастомные роли с гранулярными permissions
- Например: "Course Observer" (может видеть курс, но не может оценивать)

#### 4.5.3. Отчетность

**Типы отчетов в Canvas:**

1. **Course-level analytics:**
   - Активность студентов (page views, participation)
   - Средний балл по заданиям
   - Список студентов в риске (low engagement, low grades)

2. **Account-level reports:**
   - Количество активных курсов, студентов, преподавателей
   - Usage statistics (сколько времени проводят в системе)
   - Completion rates (процент завершивших курсы)

3. **Custom reports через Canvas Data:**
   - Canvas Data — отдельный продукт, который экспортирует все данные LMS в data warehouse (PostgreSQL / Amazon Redshift)
   - Институция может строить свои отчеты через BI-инструменты (Tableau, Power BI)

**Процесс генерации отчета:**

```
[Администратор] → Admin Panel → Reports
    ↓
Выбирает тип отчета (например, "Student Enrollment Report")
    ↓
Настраивает параметры: enrollment term, date range, включить ли удаленных студентов
    ↓
Canvas запускает: фоновую задачу генерации отчета
    ↓
Canvas уведомляет: администратора, когда отчет готов (обычно несколько минут)
    ↓
[Администратор] → Скачивает CSV-файл с отчетом
```

**Для B3 LMS важно:**
- Canvas не имеет визуального конструктора отчетов (только предзаданные шаблоны + Canvas Data для кастомных)
- B3 Platform с модулем **дашбордов и аналитики** имеет преимущество — можно строить кастомные отчеты без программирования

## 5. Что можно взять для B3 LMS

### 5.1. UX-паттерны (высокий приоритет — MUST HAVE)

**1. Dashboard как центральная точка входа**
- **Описание:** После входа студент видит Dashboard с карточками курсов, To-Do list, календарем
- **Почему важно:** Снижает когнитивную нагрузку — студенту не нужно искать, "с чего начать"
- **Реализация в B3:** Использовать **Dashboard Builder** B3 для создания персонализированного дашборда студента
- **Приоритет:** MUST HAVE

**2. Кнопка "Продолжить" вместо "Открыть курс"**
- **Описание:** Вместо перехода на главную страницу курса, кнопка ведет к следующему невыполненному заданию
- **Почему важно:** Студент экономит 2-3 клика, система "помнит" его контекст
- **Реализация в B3:** В модели данных хранить `last_accessed_item` для каждого Enrollment, кнопка "Продолжить" делает редирект на него
- **Приоритет:** MUST HAVE

**3. Прогресс-бары на карточках курсов**
- **Описание:** Визуальная индикация % завершения курса прямо на карточке
- **Почему важно:** Геймификация, мотивация ("осталось 20%, доделаю!")
- **Реализация в B3:** Вычисляемое поле `progress_percentage` = (completed_assignments / total_assignments) * 100, отображается как progress bar в Dashboard
- **Приоритет:** MUST HAVE (отсутствие этого в Canvas — частая критика пользователей)

**4. To-Do List с автоматическим формированием**
- **Описание:** Список ближайших заданий с дедлайнами, автоматически обновляется
- **Реализация в B3:** BPMN-процесс при создании Assignment автоматически добавляет задачу в To-Do пользователя; при сдаче Submission — удаляет
- **Приоритет:** MUST HAVE

**5. Календарь с интеграцией дедлайнов**
- **Описание:** Все дедлайны курсов отображаются на одном календаре, можно экспортировать в Google Calendar
- **Реализация в B3:** Виджет календаря на Dashboard, синхронизация через iCal feed (REST API endpoint)
- **Приоритет:** NICE TO HAVE (v2.0)

### 5.2. Модель данных (высокий приоритет)

**6. Разделение Enrollment на типы (роли)**
- **Описание:** Связь User ↔ Course через Enrollment с атрибутом `role` (StudentEnrollment, TeacherEnrollment, ObserverEnrollment)
- **Почему важно:** Гибкость — один пользователь может быть студентом одного курса и преподавателем другого
- **Реализация в B3:** Сущность `Запись на курс` с полем `Тип роли` (справочник: Студент, Преподаватель, Наблюдатель, Методист)
- **Приоритет:** MUST HAVE

**7. SIS ID (внешний идентификатор) для интеграции**
- **Описание:** Каждый User, Course, Enrollment имеет опциональное поле `sis_id` для связи с внешними системами (HR, ERP)
- **Почему важно:** Упрощает синхронизацию данных через API (не нужно искать пользователя по email, можно по табельному номеру)
- **Реализация в B3:** Добавить поле `Внешний ID` в сущности `Пользователь`, `Экземпляр курса`, `Запись на курс`
- **Приоритет:** MUST HAVE (особенно для корпоративных клиентов)

**8. Submission с историей версий**
- **Описание:** При multiple attempts Canvas хранит все версии сдач студента
- **Почему важно:** Прозрачность, возможность отследить эволюцию работы студента
- **Реализация в B3:** Таблица `Версии сдачи задания` (child-таблица к `Экземпляр задания`), каждая версия = отдельная запись с timestamp + файлами
- **Приоритет:** NICE TO HAVE (v2.0)

**9. Gradebook как вычисляемая сущность**
- **Описание:** Итоговая оценка не хранится отдельно, а вычисляется на лету из Submissions + настроек весов
- **Почему важно:** Избежать рассинхронизации данных, всегда актуальные оценки
- **Реализация в B3:** Вычисляемое поле в сущности `Запись на курс`: `Итоговый балл` = SUM(баллы заданий * веса категорий)
- **Приоритет:** MUST HAVE

### 5.3. Процессы (BPMN для B3)

**10. Процесс "Заявка на курс → Одобрение → Запись"**
- **Описание:** Студент подает заявку → менеджер/администратор одобряет → автоматическое создание Enrollment
- **Реализация в B3:** BPMN-процесс с задачами пользователя (User Task для одобрения), Service Task для создания записи
- **Приоритет:** MUST HAVE

**11. Процесс "Сдача задания → Уведомление преподавателю"**
- **Описание:** При создании Submission автоматически отправляется уведомление преподавателю
- **Реализация в B3:** BPMN-триггер на событие "создание Submission" → Service Task "отправить email преподавателю"
- **Приоритет:** MUST HAVE

**12. Процесс "Проверка задания → Оценка → Уведомление студенту"**
- **Описание:** Преподаватель выставляет оценку → автоматическое обновление Gradebook → уведомление студенту
- **Реализация в B3:** BPMN-процесс с User Task "Проверить задание" → Script Task "пересчитать итоговую оценку" → Service Task "отправить уведомление"
- **Приоритет:** MUST HAVE

**13. Процесс "Завершение курса → Автоматическая выдача сертификата"**
- **Описание:** При выполнении условий (все задания сданы, балл >= порога) → автоматическая генерация сертификата
- **Реализация в B3:** BPMN-процесс с Gateway (проверка условий) → Service Task "создать Экземпляр сертификата" → Service Task "сгенерировать PDF через конструктор печатных форм B3" → Service Task "отправить ссылку студенту"

## 6. Что НЕ подходит

### 6.1. Технические ограничения

**1. Canvas Data / Data Warehouse для отчетности**
- **Почему НЕ брать:** Canvas Data — это отдельный платный продукт, требующий экспорта всех данных в PostgreSQL/Redshift и настройки BI-инструментов (Tableau, Power BI). Для B3 это избыточно.
- **Альтернатива в B3:** Использовать встроенный **модуль дашбордов и аналитики B3**, который уже имеет визуальный конструктор отчетов и работает напрямую с данными платформы.

**2. Canvas Studio (видеоплатформа)**
- **Почему НЕ брать:** Canvas Studio — отдельный продукт ($$$), требует интеграции, хранилище видео, AI для субтитров. Для MVP B3 LMS это overkill.
- **Альтернатива в B3:** В v1.0 достаточно **встраивания внешних видео** (YouTube, Vimeo, VK Video) через iframe. В v2.0 можно рассмотреть интеграцию с российскими видеоплатформами (RuTube, VK Video).

**3. Mastery Paths (адаптивные траектории)**
- **Почему НЕ брать для v1.0:** Сложная логика условных переходов (if студент набрал <70% → показать доп. материалы). Требует визуального rule builder, что усложняет MVP.
- **Альтернатива в B3:** Простые prerequisites в модулях (модуль 2 открывается после модуля 1) — это достаточно для v1.0. Полноценные Mastery Paths — в v2.0+.

### 6.2. Бизнес-логика, избыточная для корпоративного контекста

**4. Canvas Commons (публичная библиотека курсов)**
- **Почему НЕ брать:** Canvas Commons — это маркетплейс открытых курсов, полезен для академий (обмен контентом между университетами), но для корпоративного B3 LMS не актуален (курсы уникальны для компании).
- **Альтернатива в B3:** Внутренняя библиотека шаблонов курсов (Шаблон курса → копирование в новый Экземпляр курса), без публичного sharing.

**5. Parent/Observer роль**
- **Почему НЕ брать:** Роль Observer (родитель наблюдает за успеваемостью ребенка) актуальна для школ K-12, но не для корпоративного обучения.
- **Исключение:** Может быть полезна, если B3 LMS будет использоваться госструктурами для школ/колледжей. В этом случае — в v2.0.

**6. Peer Review (взаимная проверка работ)**
- **Почему НЕ брать для v1.0:** Canvas поддерживает задания, где студенты проверяют работы друг друга по критериям (полезно для больших курсов, где преподаватель физически не успевает проверить все). Для корпоративного обучения B3 (небольшие группы, 10-30 студентов на поток) это избыточно.
- **Альтернатива:** В v1.0 — только преподавательская проверка. Peer Review — в backlog для v3.0+.

### 6.3. Функции, конфликтующие с B3 Platform

**7. Встроенный конструктор тестов с 15+ типами вопросов**
- **Почему НЕ брать полностью:** Canvas Quiz имеет очень сложный редактор вопросов (formula questions с переменными, matching с drag-and-drop и т.д.). Реализация этого с нуля в B3 потребует месяцев разработки.
- **Что взять:** Базовые типы вопросов для v1.0:
  - Multiple choice (один правильный ответ)
  - Multiple answers (несколько правильных ответов)
  - True/False
  - Text entry (короткий ответ)
  - Essay (длинный текст, ручная проверка)
- **Отложить на v2.0+:** Formula, matching, numerical with tolerance, fill-in-multiple-blanks.

**8. Lockdown Browser / Proctoring интеграции**
- **Почему НЕ брать для v1.0:** Системы прокторинга (Respondus, Proctorio) требуют:
  - Установку клиентского ПО (lockdown browser блокирует другие приложения)
  - Интеграцию с вебкамерами, AI для детекции списывания
  - Высокие затраты на лицензии ($$ за студента)
- **Альтернатива в B3:** В v1.0 — честность студентов + дедлайны. Для критичных сертификационных экзаменов (v2.0+) можно рассмотреть интеграцию с российскими системами прокторинга (Examus, Proctor.Online).

**9. Gamification out-of-the-box (бейджи, лидерборды)**
- **Почему НЕ брать для v1.0:** Canvas НЕ имеет встроенной геймификации, это делается через Canvas Credentials (платная интеграция). Для B3 LMS разрабатывать собственную систему бейджей в v1.0 — низкий приоритет.
- **Что взять:** Простой прогресс-бар (% завершения курса) — это уже элемент геймификации.
- **Отложить:** Бейджи, лидерборды, XP (experience points) — в v2.0+, если будет запрос от заказчиков.

### 6.4. UX-решения с критикой

**10. Отсутствие прогресс-бара на карточке курса на Dashboard**
- **Проблема:** Пользователи Canvas жалуются, что на Dashboard нет визуального индикатора прогресса курса — нужно заходить в курс, чтобы понять, сколько сделано.
- **Вывод для B3 LMS:** НЕ повторять эту ошибку — обязательно добавить прогресс-бар на карточки курсов (см. пункт 3 в разделе 5).

**11. Слишком глубокая навигация для некоторых функций**
- **Проблема:** Некоторые функции Canvas "спрятаны" глубоко в меню (например, настройка notification preferences — 4 клика от Dashboard).
- **Вывод для B3 LMS:** Часто используемые функции (профиль, уведомления, календарь, сообщения) должны быть доступны в один клик из глобальной панели.

**12. Inbox (внутренняя почта) как отдельная система**
- **Проблема:** Canvas Inbox — это отдельная система сообщений, не интегрированная с email. Студенты жалуются, что приходится проверять два места (email + Canvas Inbox).
- **Вывод для B3 LMS:** Интеграция уведомлений с email — обязательна. Внутренняя система сообщений может быть дополнением, но не заменой email.

**Конец анализа Canvas LMS**

**Основные выводы для B3 LMS:**

- Canvas демонстрирует, что **чистый UX — это конкурентное преимущество** в LMS
- Ключевые паттерны для заимствования: Dashboard, To-Do, прогресс-трекинг, SpeedGrader
- Модель данных Canvas хорошо ложится на B3 Platform (Enrollment, Submission, модули с prerequisites)
- Сильная сторона B3 над Canvas: **встроенный конструктор печатных форм для сертификатов**
- Слабая сторона Canvas для корп. сектора: платные add-ons для сертификации, отчетности, видео
- B3 Platform с модулями BPMN, Dashboard Builder, Data Model Constructor позволяет реализовать 80% функциональности Canvas без внешних интеграций
